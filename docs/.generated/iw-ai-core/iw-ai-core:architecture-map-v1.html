Executive Summary

IW AI Core is a standalone orchestration platform that drives AI-assisted development across multiple software projects. It replaces file-based tracking (markdown ID files, JSON manifests) with a PostgreSQL-backed system where the database is the single source of truth for all operational state: IDs, step statuses, batch execution, agent heartbeats, and merge queues. A single daemon process polls the database every 60 seconds, creates git worktrees for agents, and manages the full lifecycle from work item creation through merge and archive. The FastAPI dashboard on port 9900 provides the human interface across all registered projects.

Core Components

The Database (PostgreSQL on port 5433)

Every table uses project_id for multi-project isolation. Composite primary keys (project_id, id) ensure no cross-project collisions. Atomic ID allocation uses FOR UPDATE row locks — no race conditions under concurrent agent access.

Core tables:

Table Purpose
projects Registration: id, display_name, repo_root, enabled
id_sequences Atomic ID allocation: (project_id, prefix) → next_number
work_items Design doc, title, status (draft→approved→in_progress→completed|failed), phase, Tier-1 content
workflow_steps Step number, agent, status, report content (Tier 1)
step_runs Execution control plane: PID, pid_alive, heartbeat, timeout, command (for restart), exit_code
batches Batch-level status, max_parallel, auto_publish
batch_items Per-item status in a batch: pending→setting_up→executing→completed→merged|failed
daemon_events Audit trail: event_type, entity_id, message, metadata

Tier 1 storage: design docs and reports stored as TEXT in work_items and workflow_steps. Always viewable, full-text searchable via tsvector.

Tier 2 storage: full artifacts (prompts, logs, evidences) compressed to archive/<project>/<item_id>.tar.zst. Not in git. On-demand extraction to /tmp.

The Daemon

Single Python process. Single-threaded poll loop every 60 seconds. Manages all registered projects from a single process.

Responsibilities:
- Poll for approved / executing batches across all projects
- Create git worktrees via executor/worktree_setup.sh
- Launch LLM agents (opencode or claude-code) for each step
- Monitor running steps: kill -0 for aliveness, heartbeat tracking, stall detection
- Process merge queue one project at a time (squash-merge via executor/worktree_commit.sh)
- Track git status per project
- Emit events to daemon_events table
- Poll DocGenerationJob and CodeIndexJob tables

Stall detection: if a running step's last_heartbeat is more than 600 seconds old, the daemon marks it stalled.

Timeout: per-step-type configurable timeouts (default: 45 min for implementation, 30 min for code review, 10 min for lint).

The Dashboard (FastAPI on port 9900)

No React SPA. No build pipeline. Jinja2 templates + htmx for partial page updates + SSE for live notifications.

Key pages:
- /project/<project>/queue — pending work items, design docs, approve button
- /project/<project>/batches — all batches with status
- /project/<project>/batch/<batch_id> — timeline, Gantt-style execution view
- /project/<project>/item/<item_id> — step pipeline, one-click restart/kill/skip
- /system/running — all active steps across all projects with live duration counters
- /system/all-active — cross-project view of in-flight items

One-click actions (all DB-backed, daemon picks up within 60s):
- Kill: SIGTERM to the stored PID, mark step_runs.status='killed'
- Restart: re-insert step_runs row with same command + worktree_path, daemon re-launches
- Skip: mark step completed, daemon advances

The iw CLI

Bridge between LLM agents and the database. Installed via pip install -e /path/to/iw-ai-core. Available on PATH from any directory. Finds the current project by looking for .iw-orch.json up the directory tree.

Command groups:

Executor Scripts

Deterministic bash. No LLM involvement. Project-agnostic.

Script Purpose
worktree_setup.sh git worktree add, install deps, sync skills, update DB to setting_up
worktree_commit.sh Squash-merge worktree branch to main, update batch_items.merged, clean up worktree
step_executor.sh Launch LLM agent for a step, handle timeouts, parse results (legacy — daemon now handles launch directly)

Skills Distribution

Master copies live in iw-ai-core/skills/. iw sync-skills copies them to each project's .claude/skills/. Projects can override any skill by creating their own version — the sync skips overrides and records them in .iw-skills-lock.json.

Multi-Project Isolation

Isolation guarantees:
- Every table query includes project_id filter — no cross-project data leaks
- ID sequences are per-project: InnoForge I001 never collides with project-b's I001
- Worktrees are under each project's own .worktrees/ directory — filesystem-level isolation
- Merge queues are per-project: InnoForge merges never block project-b
- LLM quota is tracked globally (same account) but execution is per-project

Two-Tier Storage Model

Tier 1 (always viewable in dashboard, full-text searchable):
- work_items.design_doc_content — full markdown of design document
- workflow_steps.report_content — full report markdown per step
- work_items.summary — AI-generated 2-3 line summary for list views

Tier 2 (on-demand extraction, not in git):
- Prompts, evidences, screenshots, raw logs
- Compressed as .tar.zst in iw-ai-core/archive/<project>/
- Extracted to /tmp/iw-archive-view/<project>/<id>/ on request, TTL-cleaned after 10 min inactivity

Active work items (in-flight, being executed) stay in the project repo at ai-dev/design/active/<id>/. Agents need filesystem access during execution.

Step Runs — Execution Control Plane

Every launch (or re-launch) of a step creates a new row in step_runs:

Field Purpose Who writes
pid OS process ID — enables kill, alive-check Daemon (on launch)
pid_alive Is process still running right now? Daemon (every poll cycle via kill -0)
command Exact launch command — enables one-click restart Daemon (on launch)
worktree_path Where the agent runs Daemon (from worktree_setup)
last_heartbeat Last confirmed activity Daemon (every poll cycle)
timeout_secs Dynamic per step type, not global constant Daemon (set from step config)
error_message Human-readable failure reason Daemon or agent (via iw step-fail --reason)

Restart flow: User clicks [Restart] → dashboard inserts new step_runs row with same command + worktree_path + incremented run_number → daemon picks up pending status on next poll → re-executes with identical parameters.

Dynamic timeouts per step type:

Step type Default timeout
Implementation (Backend, API, DB) 45 min
Code Review (per-agent) 30 min
Code Review (final) 40 min
Quality Validation (lint/format) 10 min
Quality Validation (tests) 20 min
Quality Validation (browser) 15 min
QV Fix (LLM-driven) 30 min
