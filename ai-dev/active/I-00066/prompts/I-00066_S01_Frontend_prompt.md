## Requirements

### 1. Widen the modal

Edit `dashboard/static/tailwind.src.css` so that the `.oss-modal-inner`
rule (currently around lines 141-152) uses `max-width: 80vw` in place
of `max-width: 36rem`. Preserve every other property on the rule
(`background-color`, `border`, `border-radius`, `width: 100%`,
`max-height: 90vh`, `display: flex`, `flex-direction: column`,
`overflow: hidden`, `box-shadow`).

The `36rem` literal MUST NOT remain anywhere inside the `.oss-modal-inner`
block — the regression test asserts its absence.

### 2. Restyle the existing footer buttons

Update the `.modal-apply, .modal-rerun, .modal-accept` rule (around
lines 224-241) so that the buttons render with clearly visible button
affordance — without using flashy or brand-coloured backgrounds. Keep
the rule using existing CSS custom properties only. At minimum:

- A real, slightly stronger border (consider increasing from
  `1px solid var(--border)` to `1px solid var(--border)` with an
  inset highlight via `box-shadow`, or to a subtle muted-tinted
  background — pick one consistent direction; do NOT introduce a
  brand colour).
- Slightly larger padding so buttons read as buttons, not inline text
  (e.g. `0.5rem 0.875rem`).
- A `font-weight: 500` baseline (already there) and consistent
  `line-height`.
- A clearer hover state — for example a transition + slightly darker
  muted background, still without using `var(--primary)` or
  `var(--accent)` as the resting colour.

Add `.modal-preview` to the same rule list so that the (currently
hidden) Preview button gets the same styling.

### 3. Add a peer footer-button class for the footer Close button

Add a NEW rule `.modal-footer-close` to the same file that mirrors the
visual style of the other footer buttons (so all footer actions render
with the same height and shape). Do NOT modify the existing
`.modal-close` rule — the header `×` close button must keep its
existing minimal style.

The new rule must include at least:
- A real `border:` declaration (semantic test asserts `border:` text is
  present in the rule).
- A real `padding:` declaration (semantic test asserts `padding:` text
  is present).
- A hover state.

### 4. Apply the new class in the template

Edit `dashboard/templates/fragments/oss_finding_modal.html` line 74 so
the footer Close button reads:

```html
<button type="button" class="modal-footer-close modal-close">Close</button>
```

Both classes are required:
- `modal-close` keeps the existing JS click handler matching
  (`oss_finding_modal.html:335-345` checks
  `ev.target.classList.contains('modal-close')`).
- `modal-footer-close` provides the new peer-button styling.

The header `×` close button on line 11 MUST be left unchanged — it
keeps `class="modal-close"` only.

### 5. Rebuild the compiled stylesheet

Run `make css` after editing `tailwind.src.css`. Confirm
`dashboard/static/styles.css` was regenerated and that the new
`max-width: 80vw` value appears in it (the regression test asserts
this). Stage both `tailwind.src.css` and the regenerated
`styles.css` in your changes.

### 6. Manual sanity check (optional but recommended)

If a dashboard is reachable from the worktree, open it locally and
click a "..." details button on the OSS page to confirm the modal
looks correct. The QV-Browser step (S14) performs the canonical
verification inside an isolated stack — your manual check is just a
sanity check.
