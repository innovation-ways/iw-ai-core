## Description

Surfaces the `ProjectDoc` diagrams generated by F-00064 in the dashboard Code
view: a dedicated architecture diagram panel on the code index page, and a
per-module diagram panel below the module doc in the module detail view. Also
fixes a latent bug where `<div class="mermaid">` blocks in the architecture map
never render because the chat's `mermaid.js` overrides `startOnLoad: false`.

## Project Context

Read `CLAUDE.md`, `dashboard/CLAUDE.md`, and `orch/rag/CLAUDE.md`. Key patterns:
FastAPI + Jinja2 + htmx. Routers are thin. Fragment templates must NOT extend
`base.html`. CSS is prebuilt via `make css` — run it after template changes.
The brand-themed Mermaid pipeline lives in `dashboard/static/chat/mermaid.js`
as `window.iwChat.upgradeMermaidBlock` / `upgradeAllMermaidBlocks`.

## Scope

### In Scope

- New API endpoint: `GET /api/projects/{project_id}/code/modules/{slug}/diagram` — returns `code_module_diagram.html` fragment with the module's stored diagram DSL
- New fragment: `dashboard/templates/fragments/code_module_diagram.html` — `<pre data-lang="mermaid">` block + upgrade trigger + expand button + "no diagram yet" empty state
- Updated fragment: `dashboard/templates/fragments/code_module_detail.html` — adds a "Diagram" section below the doc content that htmx-loads the diagram fragment
- New fragment: `dashboard/templates/fragments/code_architecture_diagram.html` — standalone panel showing the `diagram-architecture` ProjectDoc DSL
- Updated fragment: `dashboard/templates/fragments/code_architecture_view.html` — includes the architecture diagram panel, and fixes Mermaid rendering by changing `_preprocess_mermaid` output format
- Updated router: `dashboard/routers/code_ui.py` — loads `diagram-architecture` doc on code page load
- Updated router: `dashboard/routers/code.py` — new `GET .../modules/{slug}/diagram` endpoint
- Fix: `dashboard/utils/markdown.py:_preprocess_mermaid` — change output to `<pre data-lang="mermaid"><code>` format so `upgradeAllMermaidBlocks` can pick it up (current `<div class="mermaid">` format never renders because chat JS sets `startOnLoad: false`)
- Small JS init in `code_architecture_view.html` — calls `iwChat.upgradeAllMermaidBlocks(container)` after htmx loads the fragment
- `make css` — run after any template changes to rebuild `dashboard/static/styles.css`

### Out of Scope

- QA chat diagram events (F-00066)
- Diagram regeneration button (can be a future CR)
- Showing multiple diagram types per module
- The `orch/diagram/render.py` server-side rendering pipeline (F-00064; used here as an import)

## Acceptance Criteria

### AC1: Module diagram visible in module detail view

```
Given F-00064 has stored a diagram doc with doc_id="diagram-module-{slug}"
When a user opens the module detail for that slug in the code view
Then a "Diagram" section appears below the module documentation
     containing a rendered Mermaid diagram with ELK layout and brand theming
```

### AC2: Architecture diagram visible on code index page

```
Given F-00064 has stored a diagram doc with doc_id="diagram-architecture"
When a user opens the Code tab for a project
Then an architecture diagram panel is visible in the architecture view
     containing the rendered architecture component diagram
```

### AC3: No diagram stored — graceful empty state

```
Given no diagram ProjectDoc exists for a module slug
When GET /api/projects/{id}/code/modules/{slug}/diagram is called
Then the response is a 200 with an empty-state fragment
     (not a 404 or error state)
```

### AC4: Mermaid blocks in architecture text render correctly

```
Given the architecture map content includes a ```mermaid fenced block
When the code page loads (after the _preprocess_mermaid fix)
Then the Mermaid block is rendered as a brand-themed diagram
     (not raw DSL text, not a blank div)
```

### AC5: Diagram endpoint returns 404 for unknown project/slug

```
Given a project_id or slug that does not exist in the DB
When GET /api/projects/{project_id}/code/modules/{slug}/diagram is called
Then the response is 404
```

## Boundary Behavior

| Scenario | Input/State | Expected Behavior |
|----------|-------------|-------------------|
| No diagram doc for module | `doc_id` not in `project_docs` | Fragment shows "No diagram yet" empty state; no error |
| No diagram-architecture doc | Indexing not yet run | Architecture diagram panel hidden or shows empty state |
| Module doc generating | `generating=True` on module detail | Diagram section shows "pending generation" note |
| Invalid project_id | Non-existent project | 404 from endpoint |
| Mermaid DSL has ELK frontmatter | Content starts with `---\nconfig:\n  layout: elk\n---` | Renders correctly — frontmatter is passed to upgradeMermaidBlock as-is |
| JavaScript disabled | Client has JS off | `<pre>` shows raw DSL as readable fallback |

## Invariants

1. The `code_module_diagram.html` fragment MUST NOT extend `base.html`.
2. The new endpoint MUST use `DocService` (or a direct `session.scalar`) to read `project_docs` — no direct SQL string queries.
3. The `_preprocess_mermaid` change MUST output `<pre data-lang="mermaid"><code>{dsl}</code></pre>` instead of `<div class="mermaid">{dsl}</div>`.
4. `upgradeAllMermaidBlocks` MUST be called after the architecture fragment is htmx-swapped in (via `htmx:afterSwap` event or equivalent).
5. The diagram `<pre data-lang="mermaid">` blocks must be inside a container that `upgradeAllMermaidBlocks` can query — use the fragment's root element as the scope.

## Notes

- The `_preprocess_mermaid` function in `dashboard/utils/markdown.py` is also used for rendering docs in other places (check all callers with `grep -rn "_preprocess_mermaid\|preprocess_mermaid"` before changing). If it's used outside the code architecture view in contexts that DON'T have the chat JS loaded, the change from `<div class="mermaid">` to `<pre data-lang="mermaid">` would break rendering there. Check callers carefully and apply the fix only where safe, or add a parameter to control output format.
- The `upgradeAllMermaidBlocks` function in `mermaid.js` queries for `pre[data-lang="mermaid"]`. The new fragments use this format exactly.
- Browser verification must confirm: (1) module diagram renders with ELK/brand theming, (2) architecture diagram renders, (3) empty state shows for modules without diagrams, (4) no console errors.
