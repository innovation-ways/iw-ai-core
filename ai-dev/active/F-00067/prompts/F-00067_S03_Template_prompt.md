## Context

The `iw-doc-generator` skill generates module documentation for InnoForge projects. The current `module-doc-template.md` has no "Why" section, no guidance on callout usage, and no diagram color spec. This step updates the skill templates so that future generated docs follow the new visual design standards. It also updates `iw-tech-doc-writer`'s diagram guidelines to include the canonical color palette.

**IMPORTANT**: After all file edits, run `iw skills sync` to propagate changes to all managed projects. Read `projects.toml` to understand which projects will receive the sync. Report any sync errors as non-blocking notes.

## Requirements

### 1. Update `skills/iw-doc-generator/references/module-doc-template.md`

Add a **"## Why Read This"** section immediately after the `# {Module Name}` heading (before `## Purpose`):

```markdown
## Why Read This

> [!NOTE]
> Use this document when you need to understand the internal structure of the **{Module Name}** module — what components exist, how they depend on each other, and where to add new functionality.

_Not the right doc?_ If you need API endpoint details → see the API reference. If you need to understand the overall system architecture → see the Architecture Overview.
```

Add a **"## Key Diagrams"** section after `## Architecture` and before `## Key Components`:

```markdown
## Key Diagrams

> [!TIP]
> The component diagram below shows only the major structural elements. For a full system view, see the Architecture Overview.

{Insert component diagram here — generated by the code map tool.}
```

Add a **"## Callout Usage Rules"** note in the developer notes section or as a comment in the template:

```markdown
<!-- Template convention: use callouts sparingly.
  [!NOTE]    — supplementary context that interrupts flow minimally
  [!TIP]     — shortcut or best practice worth highlighting
  [!WARNING] — behavior the reader MUST NOT miss
  [!DANGER]  — destructive or breaking behavior
-->
```

### 2. Create `skills/iw-doc-generator/references/diagram-guidelines.md`

Create a new reference file with the following content:

```markdown
# Diagram Guidelines for Doc Generator

## Semantic Color Palette (canonical)

All Mermaid diagrams generated by this skill MUST use this classDef palette:

| Class   | Role                              | Fill    | Stroke  | Text    |
|---------|-----------------------------------|---------|---------|---------|
| api     | API / router / CLI entry points   | #DBEAFE | #3B82F6 | #1E3A5F |
| data    | Database / repository / storage   | #D1FAE5 | #10B981 | #065F46 |
| worker  | Background jobs / daemon          | #FEF3C7 | #F59E0B | #78350F |
| external| External APIs / third-party       | #F3F4F6 | #9CA3AF | #374151 |
| ui      | Dashboard / frontend              | #EDE9FE | #8B5CF6 | #3B0764 |
| core    | Core orchestration / services     | #FEE2E2 | #EF4444 | #7F1D1D |

Always add this block after the graph declaration:
```
classDef api fill:#DBEAFE,stroke:#3B82F6,color:#1E3A5F
classDef data fill:#D1FAE5,stroke:#10B981,color:#065F46
classDef worker fill:#FEF3C7,stroke:#F59E0B,color:#78350F
classDef external fill:#F3F4F6,stroke:#9CA3AF,color:#374151
classDef ui fill:#EDE9FE,stroke:#8B5CF6,color:#3B0764
classDef core fill:#FEE2E2,stroke:#EF4444,color:#7F1D1D
```

## "Why" Paragraph Rule

Every diagram MUST be preceded by a 1–2 sentence paragraph answering:
- What question does this diagram answer?
- When should a developer refer to it?

Example: _"This diagram shows the internal component structure of the `auth` module. Use it when adding a new authentication provider or tracing a login request through the system."_

## Diagram Type Selection

| Need | Diagram Type | Mermaid Syntax |
|------|-------------|----------------|
| System overview | C4 Context / high-level flowchart | `flowchart TB` with subgraphs |
| Container / service map | C4 Container | `flowchart TB` |
| Module internals | Component | `flowchart LR` (left-to-right) |
| Request / event flow | Sequence | `sequenceDiagram` |
| Database schema | ER | `erDiagram` |
| State machine | State | `stateDiagram-v2` |

## Abstraction Level Rules

- **One diagram = one abstraction level.** Never mix an "AuthService" box with a `getUserById()` function in the same diagram.
- **Show structural elements only**: controllers, services, repositories, adapters, domain models.
- **Exclude**: utility classes, helpers, DTOs, config objects, private methods.
- **Maximum 12 nodes** per diagram. Group minor items if needed.

## Anti-Patterns to Avoid

| Anti-Pattern | Fix |
|---|---|
| All components in one mega-diagram | Split: one overview + per-module detail |
| Default tool colors (bright green/blue/red) | Use the canonical palette above |
| No labels on relationships | Every arrow gets a verb: "calls", "stores", "publishes" |
| Diagram without title or purpose | Always add a "Why" paragraph before the diagram |
```
