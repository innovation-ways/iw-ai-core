## Context

You are implementing the fix described in the I-00039 design document. The
Jobs page (`/project/{id}/jobs`) has two visual issues:

1. **Type column** renders coloured pills via the `type_chip` macro. The user
   wants plain text in the same standard colour as the Title column — no
   chip, no per-type colour. The `type_chip` macro is duplicated in two
   files; both copies must be deleted.
2. **Type and Status filters** are flat checkbox groups. Replace each with a
   single dropdown button that opens a popover panel of checkboxes
   (multi-select). The query-string contract (`?type=A&type=B&status=X`)
   must NOT change — the backend already accepts repeated values.

Read the design document first. Read `dashboard/CLAUDE.md` for the stack
constraints (Jinja2 + htmx + prebuilt Tailwind + minimal vanilla JS — NO JS
framework, NO new dependencies).

## Requirements

### 1. Remove `type_chip` from both files; render Type cell as plain text

`dashboard/templates/pages/project/jobs.html` lines 21–32 and
`dashboard/templates/fragments/jobs_table.html` lines 21–32 each define an
identical `type_chip` macro. Only the one in `jobs_table.html` is invoked
(line 81: `{{ type_chip(row.job_type.value) }}`). Both copies must be
**deleted**.

In `jobs_table.html`, replace the call site at line 81 with a plain text
cell that uses the same colour utility class as the Title cell. Look at the
Title cell in the same file to see what colour class it uses; match that
exactly. The simplest correct change: render the type name as plain text
inside the `<td>`, no `<span>` wrapper, no colour utility classes beyond
inheriting the table's text colour. Match the Title cell's pattern.

### 2. Build a reusable multi-select dropdown component

Create `dashboard/templates/components/multi_select.html` exporting a Jinja
macro that renders a single dropdown button + popover panel. Suggested
signature:

```jinja
{% macro multi_select(name, label, options, selected) %}
{# Renders:
   - A button (text: "{{ label }}" or "{{ label }} ({n} selected)")
   - A popover <div data-multi-select-panel="{{ name }}" hidden> containing
     <label><input type="checkbox" name="{{ name }}" value="{{ option }}"
       {% if option in selected %}checked{% endif %}> {{ option }}</label>
     for each option.
   - The whole thing wrapped in <div data-multi-select="{{ name }}">.
   - All checkboxes share name="{{ name }}" so the surrounding <form method="get">
     submission produces repeated ?name=value pairs — preserves the existing
     query-string contract. #}
{% endmacro %}
```

The macro MUST:

- Use the project's existing Tailwind utility classes (look at how other
  buttons in `dashboard/templates/` are styled — e.g. the existing `Filter`
  button at `pages/project/jobs.html:86-90`). Match that visual style.
- Use `data-multi-select="{{ name }}"` on the wrapper and
  `data-multi-select-panel="{{ name }}"` on the panel — these are the hooks
  the JS uses. The Tests step (S03) also asserts on these attributes.
- Set `aria-haspopup="listbox"` and `aria-expanded="false"` on the button.
- Render the popover panel `hidden` by default.
- Place the popover absolutely-positioned beneath the button, with a high
  enough `z-index` to overlay the table.

### 3. Wire vanilla JS for the popover behaviour

Create `dashboard/static/multi_select.js`. It must:

- Find every `[data-multi-select]` on page load.
- On button click: toggle the matching `[data-multi-select-panel]`'s `hidden`
  attribute and update the button's `aria-expanded`.
- On checkbox change: update the button label to either `"{{ label }}"` (when
  zero selected) or `"{{ label }} (N selected)"` (when N>0 selected). The
  initial label must be set on DOMContentLoaded based on the pre-checked
  count (so server-rendered `?type=A&type=B` shows `"Type (2 selected)"` on
  page load).
- On `Escape` keypress while the panel is open: close the panel and return
  focus to the button.
- On click outside the wrapper: close the panel.

Keep it under ~50 lines. No frameworks, no jQuery. Use plain DOM APIs and
event delegation.

### 4. Replace the Type and Status filter blocks on the Jobs page

In `dashboard/templates/pages/project/jobs.html`:

- DELETE the dead `type_chip` macro (lines 21–32 in the current file).
- IMPORT the new component:
  `{% from "components/multi_select.html" import multi_select %}`
- Replace the Type filter block (current lines 44–56) with:
  `{{ multi_select("type", "Type", job_types, type_filter) }}`
- Replace the Status filter block (current lines 58–70) with:
  `{{ multi_select("status", "Status", job_statuses, status_filter) }}`
- Add a `<script src="/static/multi_select.js" defer></script>` to the page
  (in an appropriate `{% block scripts %}` if `base.html` defines one, or
  at the end of `{% block content %}` if not — check `base.html` first).

The date inputs, the Filter submit button, and the Clear link MUST remain
unchanged. The surrounding `<form method="get">` MUST remain unchanged so
the query-string submission contract is preserved.
