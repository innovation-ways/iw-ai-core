# CR-00083: Performance-budget test layer — pytest-benchmark assertions with regression-alert baselines

**Type**: Change Request
**Priority**: Medium
**Reason**: Test-coverage enhancement (TESTS_ENHANCEMENT Phase 4, item 4.2). We currently have no wall-clock measurement anywhere in the suite, so a perf regression in the daemon poll loop, a slow RAG query, or a dashboard route that doubles in latency can ship to users with zero signal until a human notices.
**Created**: 2026-05-24
**Status**: Draft

## Description

Add a lightweight performance-budget test layer (`tests/perf/`) using `pytest-benchmark`. Three modules measure the daemon poll-loop iteration, one RAG query, and the p50 latency of five key dashboard routes against committed baselines, asserting `mean < budget` (or `min < budget` when jitter is high) and failing on regression > 25% vs. the committed baseline. A nightly GitHub workflow surfaces regressions to the tracker. **No production code changes** — perf budgets are observation-only. This is Phase 4 item 4.2 of `ai-dev/work/TESTS_ENHANCEMENT.md`.

## Current Behavior

The test suite has six layers today (unit, integration, dashboard, E2E, security, contract, cross-project isolation). **None** of them measure wall-clock latency. The closest signal is `make smoke-sla` (the curated `smoke` marker set has a documented <60 s prose SLA, but no per-test wall-clock assertion). There are no committed perf baselines anywhere in the repo, no `pytest-benchmark` (or equivalent) dependency, no `make test-perf*` targets, and no nightly perf workflow.

Specifically:
- `orch/daemon/main.py::Daemon._poll_cycle` runs every `IW_CORE_POLL_INTERVAL` seconds in production. Its own runtime is invisible — if a refactor doubles it, the only signal is a CPU-usage chart on the host.
- `orch/rag/qa.py::CodeQA.answer_stream` is the user-facing Q&A latency. It embeds, retrieves (LanceDB), and streams. A slow embedding model or a LanceDB index regression has no test signal.
- The five highest-traffic dashboard routes (`/`, `/project/{id}/queue`, `/project/{id}/batches`, `/project/{id}/jobs`, `/project/{id}/code`) have correctness coverage in `tests/dashboard/` and `tests/e2e/`, but no latency assertion.

## Desired Behavior

A new test layer at `tests/perf/` with three modules and committed baselines:

1. **`tests/perf/test_daemon_poll_loop.py`** — measures one `Daemon._poll_cycle()` iteration against a seeded testcontainer DB; asserts `mean < budget_ms` (budget chosen from the initial measurement + 50% headroom and frozen in code as a module-level constant). At least 5 warmup + 10 measurement rounds via `pytest-benchmark`'s rounds API.
2. **`tests/perf/test_rag_query.py`** — measures one full RAG query against a seeded LanceDB index (small in-memory fixture index, embedding model stubbed deterministically); asserts `mean < budget_s`.
3. **`tests/perf/test_dashboard_routes.py`** — measures p50 over N runs (parametrize over the five routes) using `TestClient`; asserts p50 `< budget_ms` per route.

Baseline regression detection:
- Committed baseline JSON files under `tests/perf/baselines/` (one per module, generated by `pytest-benchmark --benchmark-save=baseline`).
- The perf test step invokes pytest with `--benchmark-compare=baseline --benchmark-compare-fail=mean:25%` — a regression > 25% fails the run with the measurement diff. The 25% threshold is the **start** value (per intake decision); it can be ratcheted down as the suite stabilises.

Makefile + CI surface:
- `make test-perf` — runs all three modules.
- `make test-perf-daemon` / `make test-perf-rag` / `make test-perf-routes` — individual module entry points.
- `make test-perf-update-baseline` — operator-only target that regenerates the committed baselines. Documented as requiring a CR review before any baseline-bump commit (no automated baseline updates).
- `.github/workflows/perf-budgets.yml` — nightly cron + `workflow_dispatch`. **Not** run on PR (per-PR perf measurement is too noisy per intake). On regression, the workflow appends a follow-up entry to `ai-dev/work/TESTS_ENHANCEMENT.md` and fails the workflow.

Docs + skill registration:

- `skills/iw-ai-core-testing/SKILL.md` grows a new "Performance budgets (CR-00083, Phase 4 item 4.2)" section in §4. Synced to `.claude/skills/` via `iw sync-skills`.

## Notes

**Budget choice methodology (S01/S02 must follow)**: For each perf test, the implementer first runs the test 10 times in a quiet environment, records the mean, sets `BUDGET = mean × 1.5` (the "initial measurement + 50% headroom" rule from the design), and saves the baseline JSON. The budget is a **module-level constant**, not a parameter — operator review of any budget bump is enforced via CR (no automated `make test-perf-update-baseline` calls from CI).

**Mean vs min**: The intake budget rule says "assert `mean < budget` (or `min < budget` if jitter is high)". Each perf module's docstring must record which it uses and why. Default to `mean`. Choose `min` only when the initial 10-run sample shows σ/μ > 0.3 — and record that ratio in the docstring.

**Regression threshold**: `--benchmark-compare-fail=mean:25%` per intake. The S03 strategy-doc update must explicitly call this out as the "start" threshold that can be ratcheted down once baselines stabilise — operators should not silently relax it.

**LanceDB fixture**: The RAG perf test must NOT depend on a running Ollama listener. Stub the embedding via a deterministic function (e.g., hash the query string into a fixed-dim vector) so the test measures retrieval + ranking + result-assembly cost, not model latency. The integration RAG suite already has a "skip when no local Ollama" hook in `tests/integration/rag/` — the perf test takes the opposite stance: ALWAYS run by mocking the model.

**TestClient warmup**: FastAPI's first request loads the app, routes, and template engine; this skews p50. The dashboard perf test must perform ≥3 warmup hits per route (excluded from measurement) before measurement begins. `pytest-benchmark`'s `warmup_rounds` parameter handles this directly.

**Risk**: A perf test that's environment-sensitive (CI runner CPU varies) can become flaky. Mitigation: this CR runs perf budgets **nightly only**, never on PR; the 25% threshold gives generous tolerance for runner variance; and the workflow uses `runs-on: ubuntu-latest` consistently (no matrix). If flakiness materialises, the follow-up is to pin a specific runner SKU or to widen the threshold — both ratchet adjustments, not CR-scope additions.

**Test-only scope discipline**: This CR must NOT introduce any production-code change. If a perf measurement uncovers a real production-code regression (the budget can't be met because the code is genuinely too slow), the response is: (a) raise the budget to the new measurement and (b) file a new Incident to investigate the regression. Do NOT fix production code in this CR — the precedent set by CR-00072/73/74/75/76 (genuine pre-existing defects → xfail + Incident, never fixed in-CR) applies here equally.
