## Context

The PostgreSQL `batch_item_status` enum on the live orchestration DB is missing the
`migration_invalid` and `migration_rolled_back` labels. They exist in the Python
`BatchItemStatus` enum (added in CR-00017) but no migration was ever authored to
extend the PG enum. The daemon's worktree re-attach query crashes on every startup
with `psycopg.errors.InvalidTextRepresentation` because it binds the full Python
enum value list.

Your job is to write a single Alembic migration that adds both labels.

Read the design document first, then read the reference migration
`40af3b76e1d5_cr_00021_rebase_pipeline_phase.py` — it adds `migration_rebase_failed`
to the same enum and is the canonical pattern. Your migration is structurally a
strict subset of that one (no CHECK constraint changes, no column additions —
just two `ALTER TYPE` statements).

## Requirements

### 1. Generate a migration file

Run:

```bash
uv run alembic revision -m "I-00042 add batch_item_status labels: migration_invalid migration_rolled_back"
```

This creates a new file under `orch/db/migrations/versions/<rev>_i_00042_add_batch_item_status_labels.py`.
Do **NOT** use `--autogenerate` — there is no model change to detect; this is a hand-written
schema-only migration.

Verify the file's `down_revision` points to `c062b6bf5eb3`. If the alembic CLI picks a
different parent (because the autogenerated head differs from current), edit the
`down_revision` literal manually.

### 2. Implement `upgrade()`

Add **both** enum labels using `op.execute("ALTER TYPE batch_item_status ADD VALUE IF NOT EXISTS '<label>'")`.
The statements MUST run inside `op.get_context().autocommit_block()` because PostgreSQL
forbids `ALTER TYPE … ADD VALUE` inside the same transaction as a use-site reference.

Use `IF NOT EXISTS` so the migration is idempotent (e.g., re-running on a partially
upgraded DB is a no-op).

Match the docstring style of `40af3b76e1d5` exactly: a one-line summary, a numbered
list of deltas, and a "Reversibility" note explaining that PostgreSQL does not
support removing enum labels so `downgrade()` leaves them in place.

### 3. Implement `downgrade()`

`downgrade()` is a no-op with an explanatory comment. PostgreSQL does not support
removing an enum label without rebuilding the type, and rebuilding is unsafe when
existing rows already use the value. Document this trade-off in a comment matching
CR-00021's wording (which itself follows the CR-00019 precedent).

### 4. Do NOT modify any other files

This is a pure schema fix. No changes to `orch/db/models.py`, daemon code, or
anything else. The Python enum is already correct — we are catching PG up.
