## Requirements

### 1. Add `merge_failed` to the `BatchItemStatus` enum

Edit `orch/db/models.py`:

- Locate the `BatchItemStatus` enum class.
- Add a new value `merge_failed` (lowercase, matches the existing convention: `failed`, `migration_invalid`, `migration_rebase_failed`, etc.).
- Place it next to `migration_invalid` / `migration_rebase_failed` to keep merge-pipeline-related statuses grouped.
- Locate `TERMINAL_BATCH_ITEM_STATUSES` (or equivalent constant — search the file). Add `BatchItemStatus.merge_failed` to that set so that other terminal-status assertions still hold (it IS a terminal-from-the-merge-queue's-perspective state; the *cascade* gate is a different set, edited in S03).

### 2. Generate the Alembic migration

Run from the worktree root:

```bash
uv run alembic revision --autogenerate -m "CR-00028 add merge_failed to batch_item_status enum"
```

Then **edit** the generated file under `orch/db/migrations/versions/`:

- The autogenerated diff for an enum value addition is often empty or wrong. Replace the `upgrade()` body with:
  ```python
  op.execute("ALTER TYPE batch_item_status ADD VALUE IF NOT EXISTS 'merge_failed'")
  ```
- The `downgrade()` body MUST be a documented no-op:
  ```python
  # PostgreSQL does not natively support removing values from an enum type.
  # Acceptable because the change is additive: rolling back leaves the value
  # in the type but the application no longer writes it. Operators rolling back
  # in the presence of in-flight 'merge_failed' rows must run a manual SQL
  # update first (see CR-00028 design doc Rollback Plan).
  pass
  ```
- Set `revision`, `down_revision`, `branch_labels`, `depends_on` correctly. The `down_revision` should point at the current head (find it via `uv run alembic current` or `uv run alembic heads`).

### 3. Verify with the testcontainer migration runner

The daemon's pre-merge dry-run will validate the migration on a testcontainer. As a sanity check, run the unit-test suite to confirm no model-loading regressions:

```bash
make test-unit
```

If any test references `BatchItemStatus` enum members and now fails because of an exhaustiveness check, note them in the report (do NOT fix them — that is S03/S07 territory).
