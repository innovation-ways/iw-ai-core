## What Was Done

### 1. `orch/db/models.py` — Added `merge_failed` to `BatchItemStatus` enum

- Added `merge_failed = "merge_failed"` to the `BatchItemStatus` enum, placed alongside `migration_invalid` / `migration_rebase_failed` (merge-pipeline-related statuses grouped together)
- Added `BatchItemStatus.merge_failed` to `TERMINAL_BATCH_ITEM_STATUSES` — this is the terminal-from-merge-queue's-perspective state, so it belongs in the same set as `merged`, `failed`, `migration_invalid`, `migration_rebase_failed`, etc.

### 2. Alembic Migration — `orch/db/migrations/versions/48218f84b69f_cr_00028_add_merge_failed.py`

- **Revision**: `48218f84b69f` (head: `561ddde7f5fb`)
- **Upgrade**: `ALTER TYPE batch_item_status ADD VALUE IF NOT EXISTS 'merge_failed'`
  - The `IF NOT EXISTS` guard protects against crash-recovery replay double-application
- **Downgrade**: documented no-op (`pass`) with a comment explaining that PostgreSQL cannot drop enum values, and that operators who need to roll back while `merge_failed` rows exist must manually update those rows first (see Rollback Plan in CR-00028 design doc)
