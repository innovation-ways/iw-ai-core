## Context

You are fixing two usability defects in the dashboard "Code" page chat
panel:

1. **Submit doesn't scroll** — after the user presses Enter, the chat
   container does not scroll to the bottom, so the user's just-sent
   message is hidden when the conversation overflows.
2. **Empty Assistant bubble is too tall** — the freshly appended,
   pre-stream Assistant bubble has too much vertical space, leaving dead
   air below the "Assistant" label.

Both have detailed root cause notes in the design document. Read it
first.

## Requirements

### 1. Scroll to bottom on submit (AC1)

In `dashboard/static/chat/composer.js`, inside the `chat-send` click
handler (currently lines ~260-345), call the existing `scrollToBottom()`
helper (lines ~390-395) **after** both `appendUserBubble(question)` and
`appendAssistantBubble()`. A single call after both appends is sufficient
since `scrollToBottom` uses `#chat-scroll-anchor` which sits below all
messages.

Add a one-line comment explaining WHY (per `CLAUDE.md` "default to writing
no comments"-but-keep-non-obvious-WHY rule):

```js
// Pin the user's just-typed message in view; without this, an overflowing
// chat leaves the new bubble below the fold (I-00060).
scrollToBottom();
```

### 2. Follow-scroll while streaming, but only if user is at the bottom (AC3)

In the same submit handler, the `streamAnswer({...})` call passes an
`onToken` callback. Wrap (or extend) that callback so each token chunk
also calls `scrollToBottom()` — BUT only when the user is currently at
the bottom of the chat. Reuse the existing `IntersectionObserver` (lines
~405-412): track its latest `isIntersecting` value in a closure variable
and check it inside the wrapped `onToken`. If the user has scrolled away
(anchor not intersecting), DO NOT scroll — the existing "↓ Latest" button
is the user's way back.

Apply the same conditional follow-scroll inside `onPhase` (so phase-strip
text growing the bubble doesn't push the answer below the fold either)
and `onDone` (final layout settle).

Do NOT remove or weaken the existing observer — the floating button must
keep working exactly as before.

### 3. Tighten the empty Assistant bubble (AC2)

The root cause is a hand-written rule in `dashboard/static/chat.css:3`:

```css
#chat-messages > article[data-role="assistant"]:last-child { min-height: 50dvh; }
```

This forces every empty assistant bubble that is the last message to be
at least 50% of the dynamic viewport height — that's the dead air the
user sees before tokens arrive.

**Required edit:** delete that rule outright. Do NOT replace it with a
smaller min-height — once the AC1/AC3 follow-scroll fixes from §1/§2
are in place, the rule's original intent (keep the streaming answer in
view) is served by active scrolling, so the rule becomes redundant.

After the edit, open the dashboard in a real browser via `playwright-cli`
and confirm the empty bubble height is ≤ 48px:

```bash
playwright-cli kill-all
playwright-cli open "$IW_BROWSER_BASE_URL"
# … log in, navigate to a project's Code page with the index built …
playwright-cli snapshot
playwright-cli fill <chat-input-ref> "explain this module"
playwright-cli click <chat-send-ref>
# Immediately:
playwright-cli evaluate "(() => {
  const a = document.querySelector('article[data-role=\"assistant\"]:last-of-type');
  return { h: a.getBoundingClientRect().height, html: a.outerHTML };
})()"
```

If after deleting the rule the bubble is **still** > 48px tall, an
additional contributor exists. Investigate (likely candidates: a
renderer-injected `phase-strip` at `render.js:330-336` rendering with
empty content, or another stray min-height/padding rule). Document
what you find in the report and apply the smallest correct fix:
- Defer creating the phase-strip until the first phase event has actual
  text content.
- Add `.phase-strip:empty { display: none; }` to `chat.css` to collapse
  any empty phase placeholder.

`dashboard/static/chat.css` is hand-written and committed directly — do
NOT run `make css`; that target does not regenerate it. The Tailwind
output `dashboard/static/styles.css` is unrelated to this fix.
