## Requirements

### 1. Generate the Alembic revision

Run `uv run alembic revision -m "add_pi_runtime_options"` (no `--autogenerate` — this is a data-only migration; autogenerate would detect no schema diff and produce an empty body). The revision file will be created with a fresh `revision` id and a `down_revision` pointing at the current head. Do not hand-pick either id.

### 2. Implement the `upgrade()` body

Copy the structure of `orch/db/migrations/versions/d1e2f3gpt53c_add_gpt_5_3_codex_runtime_option.py`. Specifically:

a) **Realign the SERIAL sequence** before inserting, using `pg_get_serial_sequence('agent_runtime_options', 'id')` with `setval(..., GREATEST(MAX(id), 1), true)`. This is required because F-00081 seeded rows 1–5 with explicit ids via `bulk_insert` without bumping the sequence; per-worktree DBs restored from `pg_dump` need the same realignment.

b) **Insert two rows** with `ON CONFLICT ON CONSTRAINT uq_agent_runtime_options_cli_model DO UPDATE` for idempotency:

| cli_tool | model | cli_label | model_label | display_name | is_default | enabled | sort_order |
|----------|-------|-----------|-------------|--------------|------------|---------|------------|
| `pi` | `minimax/MiniMax-M2.7` | `Pi` | `MiniMax 2.7` | `Pi + MiniMax 2.7` | `false` | `true` | `25` |
| `pi` | `openai/gpt-5.3-codex` | `Pi` | `GPT-5.3 Codex` | `Pi + GPT-5.3 Codex` | `false` | `true` | `26` |

Use raw `op.execute(...)` SQL, not `op.bulk_insert(...)` — `bulk_insert` does not support `ON CONFLICT`. Use parameter-free SQL (string-literal values are fine here; no user input).

### 3. Implement the `downgrade()` body

Two `DELETE FROM agent_runtime_options WHERE cli_tool = 'pi' AND model = '<model>'` statements, one per inserted row. Do NOT include a `WHERE cli_tool = 'pi'` blanket delete — be explicit about each `model` so a future Pi row with a different model isn't accidentally swept.

### 4. Update column comments in `orch/db/models.py`

- Line ~815: `StepRun.cli_tool.comment` changes from `"LLM CLI tool used: 'opencode' or 'claude'"` to `"LLM CLI tool used: 'opencode', 'claude', or 'pi'"`.
- Line ~1082 surrounding context: the `Batch.cli_tool` column does not have an explicit `comment=` today (it's just `mapped_column(Text, nullable=False, server_default=text("'opencode'"))`). Add `comment="LLM CLI tool used: 'opencode', 'claude', or 'pi'"` to it. The `server_default` of `'opencode'` stays unchanged.

Then `alembic revision --autogenerate -m "add_pi_runtime_options_column_comments"` would detect the column-comment diff, BUT the design calls for a **single migration**. So instead: add `op.alter_column(... existing_comment=..., comment=...)` calls into the same revision file's `upgrade()` body (and the reverse into `downgrade()`). Reference the existing `op.alter_column` patterns in `orch/db/migrations/versions/` for the exact syntax — there are several examples of comment-only changes in the history.

### 5. Verify with `make migration-check`

Run `make migration-check` locally. This must report green for:
- `alembic upgrade base → head` succeeds
- Resulting schema matches `Base.metadata.create_all()` (parity check)
- `downgrade base → upgrade head` round-trip succeeds

If `make migration-check` fails, fix the migration file (or `orch/db/models.py` comment string) so both halves agree. Do NOT report `tests_passed: true` while this gate is red — downstream agents will inherit a wrong schema and waste fix-cycles diagnosing the symptom.
