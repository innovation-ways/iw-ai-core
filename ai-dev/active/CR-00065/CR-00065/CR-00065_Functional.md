# CR-00065 — Functional Design

## ⛔ Docker is off-limits

(Standard policy. Testcontainer fixtures in tests are exempt.)

## ⛔ Migrations: agents generate, daemon applies

This item adds one new nullable column to the step runs table. The migration file is generated by the database agent and applied by the daemon during the merge pipeline — never by the agent directly.

## Why

When an agent step runs using the Pi runtime, all conversational output — what the agent thought, which files it read, which commands it ran — is stored in Pi's own session files, not in the standard log. The platform captures an empty log file for every Pi run. This means that when a step crashes or behaves unexpectedly, operators cannot see what the agent did without manually locating and parsing Pi's internal session files. The CR-00064 S01 crash (35 minutes of work, 0 bytes in the log) was the direct trigger. This change gives operators full session visibility from within the platform, for all three supported agent runtimes.

## What Changed (for the User)

- The step table inside any item detail page now has a "Logs" icon button to the right of each step's status badge.
- Clicking the button opens a popup showing the agent's session content for that step's most recent run.
- For Pi runs, the popup shows a structured, human-readable view: assistant messages, tool invocations (command summaries), and any error events — not raw technical data.
- For Claude and OpenCode runs, the popup shows the captured terminal log output.
- While a step is actively running, the popup refreshes every three seconds so operators can watch progress in near-real time.
- For completed or failed steps, the popup shows the final recorded state of the session.

## How It Behaves

When a user opens an item and looks at the step pipeline table, each step row that has been executed at least once shows a small log icon. Clicking it opens a floating panel anchored to that row. The panel immediately loads the log content for the most recent run. If the step is still running, a subtle indicator shows that the view is live and the content refreshes automatically. If the step ran with Pi, the content is presented as a conversation — assistant responses are shown as readable text, tool calls are summarised (for example, "bash: make test-unit"), and thinking blocks are collapsed by default but can be expanded. Context compaction points (where Pi reset its context mid-run) appear as a visible divider. If the step ran with Claude or OpenCode, the content is the raw terminal output, stripped of colour codes. The popup can be dismissed by clicking outside it or pressing Escape.

For steps that crashed (for example, due to a context window overflow), the popup shows everything the agent produced up to the point of failure, making post-mortem analysis possible without leaving the platform.

## Out of Scope

- Searching or filtering log content within the popup.
- Showing logs for all past runs of a step (only the most recent run is shown; earlier runs remain accessible from the existing Logs tab).
