The PostgreSQL enum name is `step_type` (declared via `_step_type_col = SAEnum(StepType, name="step_type", create_type=True)` further down in the same file) — your migration must add the new value to that enum.

### 2. Create the Alembic migration

Use `alembic revision --autogenerate -m "Add self_assess to step_type enum"` to scaffold the migration file. **Do NOT run `alembic upgrade`** — agents only write the migration; the daemon applies it. Inspect the autogenerated `upgrade()` function:

- It will likely contain `op.execute("ALTER TYPE step_type ADD VALUE 'self_assess'")` or you may need to write that explicitly. Check existing migrations in `orch/db/migrations/versions/` that extended enums (search for `ADD VALUE` in that directory) to match the project's exact pattern.
- The migration MUST run outside a transaction for `ALTER TYPE ... ADD VALUE` (PostgreSQL requirement). If existing migrations use a `with op.get_context().autocommit_block():` wrapper or a `transactional = False` declaration, follow the same pattern.
- The `downgrade()` function: PostgreSQL has no `ALTER TYPE ... DROP VALUE`. Follow the convention used by previous enum-extending migrations in this repo (most likely a no-op downgrade or a `raise NotImplementedError`).

### 3. Verify the migration in a testcontainer

Run `make test-integration` (or just the database/migration tests) — the testcontainer will apply your migration as part of fixture setup. Confirm:

- The migration applies cleanly to a fresh container.
- Inserting a `WorkflowStep` row with `step_type=StepType.self_assess` succeeds.
- Existing tests pass without modification (no row in the DB references the new value yet, so no behavioral break).

If a test references `StepType` in an exhaustive switch/match and complains about a missing case, that test belongs to S03 or later — flag it as a blocker rather than fixing it here.
