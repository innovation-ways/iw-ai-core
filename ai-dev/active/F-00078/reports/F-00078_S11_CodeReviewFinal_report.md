### Sidecar Path Convention

**Skill output** (S07, `skills/iw-item-analyze/SKILL.md`):
- Report: `ai-dev/work/<ID>/reports/<ID>_self_assess_report.md`
- JSON: `ai-dev/work/<ID>/reports/<ID>_self_assess_findings.json`

**Sidecar resolver** (`orch/self_assess.py:findings_path_for`):
- `<stem>_self_assess_report.md` → `<stem>_self_assess_findings.json`
- `<stem>_self_assess.md` → `<stem>_self_assess_findings.json` ✅

**Dashboard loader** (`orch/daemon/execution_report.py:_load_self_assessment`):
- Uses `findings_path_for(report_path)` to derive JSON path ✅

**Path example trace**:
- Report: `ai-dev/work/F-00078/reports/F-00078_S01_self_assess_report.md`
- Derived JSON: `ai-dev/work/F-00078/reports/F-00078_S01_self_assess_findings.json`
- Skill writes: `ai-dev/work/F-00078/reports/F-00078_self_assess_findings.json`

**⚠️ INCONSISTENCY**: The skill writes to `<ID>_self_assess_findings.json` (stem only), but the report written by the step executor follows the pattern `<ID>_S{NN}_self_assess_report.md`. When the step is S01, the report is `F-00078_S01_self_assess_report.md`, so the derived JSON path is `F-00078_S01_self_assess_findings.json`, but the skill writes `F-00078_self_assess_findings.json`.

This is a **HIGH** cross-cutting inconsistency. The skill writes to a path that does not match what `findings_path_for` derives from the actual report path.

## Findings

### CRITICAL (Must Fix)

None.

### HIGH

1. **Sidecar path mismatch between skill output and `findings_path_for` convention**
   - **File**: `skills/iw-item-analyze/SKILL.md` (Phase 3 Output) + `orch/self_assess.py:findings_path_for`
   - **Description**: The skill writes findings JSON to `<ID>_self_assess_findings.json` (stem-only, no step suffix). The `findings_path_for` convention derives the JSON path from the report path: for `F-00078_S01_self_assess_report.md`, it derives `F-00078_S01_self_assess_findings.json`. These paths don't match when the step ID differs from the item ID pattern.
   - **Suggestion**: The skill should write to the step-specific path that matches the report. The report path follows the pattern `<ID>_<STEP>_self_assess_report.md`, so the JSON should be `<ID>_<STEP>_self_assess_findings.json`. Update the skill's Phase 3 output to construct the path from the step context (or document the convention more precisely). Alternatively, standardize both to use `<ID>_self_assess_findings.json` regardless of step.
   - **Cross-cutting**: Affects S07 (skill) and S03 (convention helper).

### MEDIUM (Fixable)

1. **Design skills don't set explicit `step_type` in manifest**
   - **File**: `skills/iw-new-feature/SKILL.md`, `skills/iw-new-cr/SKILL.md`, `skills/iw-new-incident/SKILL.md`
   - **Description**: The design skills inject `{ "agent": "self-assess-impl" }` without an explicit `"step_type": "self_assess"` field. The `agent_to_step_type("self-assess-impl")` inference works (pattern matches), but an explicit field is more robust and self-documenting.
   - **Suggestion**: Add `"step_type": "self_assess"` to the injected step dict in all three design skills. This costs nothing and makes the manifest authoritative.

### MEDIUM (Suggestion)

1. **Coverage claim discrepancy** (noted in S10): S09 report claims `orch/self_assess.py` coverage ≥ 90%, actual is 81%. Missed lines are defensive `isinstance` forward-compat branches — acceptable, but report should reflect actual coverage.

2. **`iw-new-incident` JSON snippet** (pre-existing, noted in S08): The skill body shows a JSON fragment starting mid-object at line 379. Not introduced by S07; clean-up is optional.
