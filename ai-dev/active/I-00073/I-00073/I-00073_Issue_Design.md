# I-00073: iw step-done/step-fail crash with UndefinedColumn when worktree ORM adds columns to step_runs/work_items

**Type**: Issue
**Severity**: High
**Created**: 2026-05-07
**Reported By**: Operator (observed during F-00079 S01 execution)
**Status**: Draft

---

## ⛔ Docker is off-limits

Standard policy. Testcontainer fixtures in tests are exempt — the integration regression test for this incident relies on the standard postgres testcontainer fixture in `tests/conftest.py`.

## ⛔ Migrations: agents generate, daemon applies

Standard policy. **This incident does NOT add or modify migrations.** The whole point of the fix is to make the agent-facing CLI tolerate the gap between an in-worktree ORM that has unmerged column additions and the live orchestration DB whose schema is still at the prior head. Adding a migration would defeat the test scenario.

## Description

When an in-flight feature in a worktree adds new columns to `step_runs` or `work_items` (via an ORM model edit + an unapplied migration file), every agent-facing `iw` command that loads those models with a full ORM `SELECT *` crashes against the live orchestration DB on port 5433 with `psycopg.errors.UndefinedColumn`. The agent finishes its real work but cannot record completion via `iw step-done` (or surface the error via `iw step-fail`); the daemon then marks the step as "Process exited without reporting completion (PID dead)" and the workflow stalls. Concrete reproduction: F-00079 S01 ran twice (run1 ~6 min, run2 ~7 min), both times failed identically because `iw step-done`'s `select(StepRun)` SELECTed `step_runs.diff_text` and `step_runs.diff_summary` — columns the worktree's ORM had but the live DB did not.

## Project Context

Read `CLAUDE.md` and `orch/CLAUDE.md` for architecture, conventions, and hard rules. The relevant constraints are:

- Agent constraint **R2** (`docs/IW_AI_Core_Agent_Constraints.md`): agents write migration FILES; only the daemon applies migrations. The drift between worktree ORM and live DB is intentional and structural — it cannot be eliminated, so the CLI must tolerate it.
- The `iw` CLI is the **agent-to-DB bridge** (`orch/CLAUDE.md`): every step-lifecycle command (`step-start`, `step-done`, `step-fail`, `step-restart`, `step-skip`, `step-kill`, `item-status`) executes inside the worktree's Python environment, importing the worktree's `orch.db.models` — so any feature that adds columns to orchestration tables changes the SELECT shape these commands emit.
- The daemon itself is **not** affected, because it always runs from `main` where ORM and DB are kept in sync by the merge pipeline.

## Steps to Reproduce

1. Approve and run a feature whose Database step adds a nullable column to `step_runs` (e.g., F-00079 added `diff_text` and `diff_summary`).
2. The Database agent writes the migration file and updates `orch/db/models.py` in the worktree, then runs preflight gates successfully.
3. The agent calls `uv run iw step-done F-00079 --step S01 --report …` from inside the worktree.

**Expected**: `iw step-done` records the step as completed in the orchestration DB and exits 0; the daemon advances to S02.

**Actual**: `iw step-done` exits with `Database error: (psycopg.errors.UndefinedColumn) column step_runs.diff_text does not exist` (the SELECT generated by `select(StepRun)` enumerates every Mapped column on the in-worktree model). The agent process exits non-zero, the daemon's PID/heartbeat watcher catches it as "Process exited without reporting completion (PID dead)", and the step is marked failed despite all the work being done correctly. The fix-cycle restarts S01, which fails identically on the very same call. After 5 fix cycles the workflow stalls indefinitely.

## Root Cause Analysis

`orch/cli/step_commands.py` and `orch/cli/item_commands.py` issue full ORM `select(Model)` queries (and `session.get(Model, key)` lookups) against the live orch DB. SQLAlchemy expands both shapes into `SELECT model.col1, model.col2, ..., model.colN` enumerating every `Mapped[]` attribute on the class. When the in-worktree class has columns that the live DB has not yet acquired, every such SELECT raises `UndefinedColumn`.

Two query shapes are equally affected:

1. `select(Model).where(...)` — explicit ORM SELECT.
2. `session.get(Model, key)` — primary-key load that ALSO emits a full-column SELECT when the entity is not in the identity map. Replacing this is slightly different: rewrite as `session.execute(select(Model).options(load_only(...)).where(Model.pk1 == k1, Model.pk2 == k2)).scalar_one_or_none()`.

## Test to Reproduce

The reproduction test must fail against the pre-fix `orch/cli/step_commands.py` and pass against the post-fix version. The trick is to simulate the worktree-vs-live drift inside a single test process: spin up a postgres testcontainer, run `Base.metadata.create_all` (which gives the testcontainer the *current* full schema as declared by the in-process ORM), then drop a specific column from `step_runs` to simulate "the in-process ORM has a column the live DB does not yet have", then invoke `iw step-done` against that DB and assert success.

```python
# tests/integration/cli/test_step_commands_drift.py
import subprocess
from sqlalchemy import text

def test_step_done_tolerates_missing_step_runs_column(
    pg_container_url, monkeypatch, tmp_path
):
    """RED before fix: step-done crashes with UndefinedColumn.
    GREEN after fix: step-done succeeds even when the live DB lacks a column
    that the in-process ORM declares (simulates worktree-vs-orch-DB drift).
    """
    # Arrange: build a DB whose step_runs table is missing diff_text.
    # The in-process ORM still declares diff_text — that is the drift.
    _bootstrap_orch_db(pg_container_url)  # create_all + FTS DDL
    _drop_column(pg_container_url, "step_runs", "diff_text")
    _seed_running_step(pg_container_url, project_id="iw-ai-core",
                       item_id="F-99999", step_id="S01")

    # Act: invoke iw step-done as a subprocess against this drifted DB.
    env = {**os.environ, **_db_env_for(pg_container_url)}
    result = subprocess.run(
        ["uv", "run", "iw", "step-done", "F-99999", "--step", "S01"],
        capture_output=True, text=True, env=env,
    )

    # Assert (semantic correctness — not just exit code):
    # 1. Command succeeded
    assert result.returncode == 0, f"stderr: {result.stderr}"
    # 2. The error message that fired before the fix is absent
    assert "UndefinedColumn" not in result.stderr
    assert "step_runs.diff_text" not in result.stderr
    # 3. The step actually advanced in the DB (not just a silent no-op)
    assert _step_status(pg_container_url, "F-99999", "S01") == "completed"
```

## Regression Prevention

- **Pinned column sets per model in the CLI layer** — once narrowed, the SELECTs no longer change shape when an unrelated feature adds a column to `step_runs` / `work_items` / `workflow_steps`. New columns the CLI legitimately needs to read or write must be added to the pinned set explicitly, and that change goes through normal review.
- **Doctstring + `docs/IW_AI_Core_Agent_Constraints.md` note** documenting the rule, so the next contributor doesn't "simplify" the projected selects back into `select(StepRun)` thinking it's cleaner.
- **Regression test** that simulates schema drift, run on every CI build via `make test-integration` — any future regression that re-introduces a full ORM SELECT in the agent-facing CLI surfaces as a test failure.

## Notes

- **Operator workaround already applied for F-00079**: the migration `1713bc13a11d` was applied out-of-band to the orch DB on 2026-05-07; the migration file was also forward-copied into `main` so alembic head matches the DB. F-00079 has been resumed (S01 restart pending). Once this incident lands, the same recovery dance will not be needed for future features.
- **Why option (a) `load_only()` and not the alternatives**: option (b) "wrap with UndefinedColumn fallback" is too magical — it would silently mask real schema bugs. Option (c) "thin pinned core schema" (separate ORM classes for the CLI) is verbose and duplicates Mapped declarations. `load_only()` is a one-line change per callsite, idiomatic SQLAlchemy 2.0, and reads naturally.
- **Two query shapes, one bug class**: `select(Model).where(...)` and `session.get(Model, key)` BOTH emit full-column SELECTs and BOTH are sensitive to worktree-vs-live schema drift. The fix must rewrite both. `session.get(Model, key)` becomes `session.execute(select(Model).options(load_only(...)).where(<pk equality>)).scalar_one_or_none()`. The "CLI resilience to in-flight schema drift" subsection added to `docs/IW_AI_Core_Agent_Constraints.md` must call out BOTH shapes so a future contributor doesn't reintroduce a `session.get` thinking it's safe by virtue of being a primary-key lookup.
- **Daemon callsites intentionally untouched**: see AC3. Consistency with `main` is guaranteed for the daemon process by the merge pipeline; narrowing daemon SELECTs would be churn.
- **The `step-restart` callsite at line 649-654 is already correct** (`select(StepRun.run_number)`). Use it as the reference pattern.
